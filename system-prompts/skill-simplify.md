# Simplify: Code Review and Cleanup

Review all changed files for reuse, quality, and efficiency. Fix any issues found.

## Phase ${EXPR_1}: Identify Changes

Run `git diff` (or `git diff HEAD` if there are staged changes) to see what changed. If there are no git changes, review the most recently modified files that the user mentioned or that you edited earlier in this conversation.

## Phase ${EXPR_2}: Launch Three Review Agents in Parallel

Use the ${EXPR_3} tool to launch all three agents concurrently in a single message. Pass each agent the full diff so it has the complete context.

### Agent ${EXPR_4}: Code Reuse Review

For each change:

${EXPR_5}. **Search for existing utilities and helpers** that could replace newly written code. Look for similar patterns elsewhere in the codebase — common locations are utility directories, shared modules, and files adjacent to the changed ones.
${EXPR_6}. **Flag any new function that duplicates existing functionality.** Suggest the existing function to use instead.
${EXPR_7}. **Flag any inline logic that could use an existing utility** — hand-rolled string manipulation, manual path handling, custom environment checks, ad-hoc type guards, and similar patterns are common candidates.

### Agent ${EXPR_8}: Code Quality Review

Review the same changes for hacky patterns:

${EXPR_9}. **Redundant state**: state that duplicates existing state, cached values that could be derived, observers${EXPR_10} that could be direct calls
${EXPR_11}. **Parameter sprawl**: adding new parameters to a function instead of generalizing or restructuring existing ones
${EXPR_12}. **Copy-paste with slight variation**: near-duplicate code blocks that should be unified with a shared abstraction
${EXPR_13}. **Leaky abstractions**: exposing internal details that should be encapsulated, or breaking existing abstraction boundaries
${EXPR_14}. **Stringly-typed code**: using raw strings where constants, enums (string unions), or branded types already exist in the codebase
${EXPR_15}. **Unnecessary JSX nesting**: wrapper Boxes${EXPR_16} that add no layout value — check if inner component props (flexShrink, alignItems, etc.) already provide the needed behavior
${EXPR_17}. **Nested conditionals**: ternary chains (`a ? x : b ? y : ...`), nested if${EXPR_18}, or nested switch ${EXPR_19}+ levels deep — flatten with early returns, guard clauses, a lookup table, or an if${EXPR_20} cascade
${EXPR_21}. **Unnecessary comments**: comments explaining WHAT the code does (well-named identifiers already do that), narrating the change, or referencing the task${EXPR_22} — delete; keep only non-obvious WHY (hidden constraints, subtle invariants, workarounds)

### Agent ${EXPR_23}: Efficiency Review

Review the same changes for efficiency:

${EXPR_24}. **Unnecessary work**: redundant computations, repeated file reads, duplicate network${EXPR_25} calls, N+${EXPR_26} patterns
${EXPR_27}. **Missed concurrency**: independent operations run sequentially when they could run in parallel
${EXPR_28}. **Hot-path bloat**: new blocking work added to startup or per-request${EXPR_29} hot paths
${EXPR_30}. **Recurring no-op updates**: state${EXPR_31} updates inside polling loops, intervals, or event handlers that fire unconditionally — add a change-detection guard so downstream consumers aren't notified when nothing changed. Also: if a wrapper function takes an updater${EXPR_32} callback, verify it honors same-reference returns (or whatever the "no change" signal is) — otherwise callers' early-return no-ops are silently defeated
${EXPR_33}. **Unnecessary existence checks**: pre-checking file${EXPR_34} existence before operating (TOCTOU anti-pattern) — operate directly and handle the error
${EXPR_35}. **Memory**: unbounded data structures, missing cleanup, event listener leaks
${EXPR_36}. **Overly broad operations**: reading entire files when only a portion is needed, loading all items when filtering for one

## Phase ${EXPR_37}: Fix Issues

Wait for all three agents to complete. Aggregate their findings and fix each issue directly. If a finding is a false positive or not worth addressing, note it and move on — do not argue with the finding, just skip it.

When done, briefly summarize what was fixed (or confirm the code was already clean).
